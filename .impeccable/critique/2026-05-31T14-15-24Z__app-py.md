#### Priority Issues

**[P1] Desktop layout breaks under real production data**

**Why it matters:** Operators need to scan quickly under pressure. Wrapped nav labels, split KPI values, tall table rows, and dense columns turn high-volume review into visual labor.

**Fix:** Reduce the first Overview KPI row from 6 to 4 decision metrics. Make the video table columns intentional: Video, Status, Step, Progress, Attention, Action. Move Runs, Redos, Started, Completed, and Duration into row detail. Widen the rail or switch to an icon rail with tooltips at constrained widths.

**Suggested command:** `impeccable layout`

**[P1] Navigation state can become misleading**

**Why it matters:** Live inspection showed a selected `Scores` nav state while Overview content remained visible for 30+ seconds. That breaks trust: the UI says one thing while showing another.

**Fix:** Do not visually commit the active nav state until content is ready, or show an explicit loading shell such as "Loading Scores..." with skeleton rows. Cache heavy Scores/Compliance loads behind a lightweight shell so navigation feels immediate and honest.

**Suggested command:** `impeccable harden`

**[P1] System status semantics are misleading**

**Why it matters:** A green dot next to CPU/RAM/Disk rows implies health even when values may be critical. This violates the Status Redundancy Rule and can train operators to ignore the status panel.

**Fix:** Replace always-green system dots with thresholded labels and icon/color pairs: `CPU Critical 100%`, `RAM High 90%`, `Disk Low 0.3 TB free`, `GPU Unavailable`. Use green only for genuinely healthy states.

**Suggested command:** `impeccable clarify`

**[P2] Incident view lacks an obvious next action**

**Why it matters:** The stalled queue alert diagnoses the problem but leaves the operator to infer the action: continue, stop, rescan, inspect, clear stale state, or restart a worker.

**Fix:** Add a compact `Recommended next step` block inside the queue attention panel. Give it one primary action and one secondary drill-down, for example `Continue queue` plus `Inspect stalled stage`.

**Suggested command:** `impeccable onboard`

**[P2] Accessibility and component consistency need hardening**

**Why it matters:** Icon-only refresh/settings controls, small muted text, collapsed labels, and side-accent score rows weaken keyboard and screen-reader confidence.

**Fix:** Add visible text/tooltips and ARIA labels for icon controls, make focus rings explicit across custom controls, run a contrast pass on muted text over translucent surfaces, and replace thick side accents with full-border or background selected states.

**Suggested command:** `impeccable harden`

#### Persona Red Flags

The dashboard gives filters, export, pagination, and queue controls, but no obvious keyboard shortcuts, no batch recovery path, and no fast "jump to the stuck item" affordance. Dense tables are present, but not yet efficient because key columns compete with historical metadata.

**Sam (Accessibility-Dependent User):** Sam depends on keyboard flow, labels, and unambiguous state. The green system dots can misrepresent health, refresh/settings are icon-first, some labels are visually collapsed, and active nav/content mismatch is disorienting. The core status rule is right in principle, but not consistently enforced.

**Rina (PROYA Operator/Editor):** Rina is monitoring a long-running production queue. The dashboard tells her the queue is stalled, but does not say what to do next. During a live production day, she needs recovery guidance above charts and detailed history.
